# Static deselect list for the PyVista test gate (ci/run-pyvista.sh).
#
# One pytest nodeid per line. `#` starts a comment; blank lines ignored. Each
# entry becomes a `--deselect <nodeid>` passed to BOTH the core and plotting
# pytest invocations (an unmatched nodeid is a no-op for the suite it doesn't
# belong to). Only the failing PARAMETRIZATIONS are listed, so passing params of
# the same test keep running.
#
# Handled elsewhere, do NOT duplicate here:
#   * needs_download network-flaky tests   -> skipped via `-m "not needs_download"`
#   * snake_case + dir() property tests     -> deselected at runtime by the script
#                                             only when the wheel disables
#                                             VTK_DISABLE_PYTHON_PROPERTIES
#
# Each entry below was confirmed by running the test against BOTH stock vtk==9.6.2
# and the cvista wheel on the SAME host/Mesa, isolating cvista-vs-stock from
# environment drift.

# --- REAL cvista divergences from stock 9.6.2 (tracked; remove when fixed) -----
# These FAIL on cvista but PASS on stock 9.6.2 on the same machine. Tracking
# issues filed on pyvista/cvista; fix each, then delete its line here.
tests/core/test_helpers.py::test_to_from_trimesh_points_faces                   # regular_faces not zero-copy (#118)
tests/plotting/test_plotting.py::test_export_gltf                               # glTF export render divergence, same-Mesa (#119)

# --- abi3 heap-type artifact (unavoidable; same family as bitexact __flags__) -
# cvista's abi3 heap types expose __module__/__doc__ in vars(<wrapped type>);
# stock's static types do not, so this introspection test sees extra entries.
tests/core/test_dataobject_filters.py::test_cell_status

# --- image-cache drift (NOT cvista: stock 9.6.2 fails identically) -------------
# stock 9.6.2 on the same Mesa also exceeds the threshold (1474 vs cvista 1466),
# so the committed image_cache was generated against a different VTK/Mesa.
tests/plotting/test_plotting.py::test_set_environment_texture_cubemap[True]

# --- by-design consequence of cvista int32-default storage (won't fix) --------
# cvista defaults topology/connectivity arrays to int32 (stock uses int64). The
# test asserts compressed_size < uncompressed_size / 4. cvista's compressed output
# is actually SMALLER than stock's (e.g. sphere: 21595 vs 22671 bytes), but its
# UNCOMPRESSED baseline is far smaller too (65199 vs 101039) because int32 halves
# the connectivity/offset arrays and stock's int64 arrays are full of zero
# high-bytes that compress almost for free. So cvista's *ratio* is ~3x vs stock's
# ~4.5x even though the absolute file is smaller. The 4x threshold assumes int64
# padding; there is no fix that keeps the int32 storage win. See #117.
tests/core/test_dataobject.py::test_save_compression                            # int32-default lowers compression RATIO (smaller file); by design (#117)

# --- intentional cvista build exclusion (won't fix) ---------------------------
# cvista's module closure excludes IOXdmf2: the vendored ThirdParty/xdmf2 fails to
# compile on modern libc++ (<strstream> removed) and only backs XdmfReader.
tests/core/test_reader.py::test_xdmf_reader

# --- upstream PyVista issues at the pinned ref (not cvista) --------------------
